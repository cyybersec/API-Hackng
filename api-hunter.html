✅ All checks completed for BOLA!
01 — Recon & Mapping
Map all API endpoints from Burp Suite history after using the app normally easy
Identify all endpoints that take an ID parameter — user ID, order ID, account ID, file ID, etc. easy
Check both URL path IDs (/api/users/123) and query params (?user_id=123) and request body IDs easy
Note all object types visible: users, orders, invoices, files, messages, payments, reports easy
Scrape JS files for hidden API endpoints not visible in the UI using LinkFinder or manually medium
Check public API documentation for all resource types and their expected parameters easy
02 — Basic BOLA Testing
Create two separate accounts — Attacker and Victim. Note both user IDs and tokens clearly easy
As Attacker, access Victim's resource by replacing the ID in the request with Victim's ID critical
Test GET requests first — /api/orders/VICTIM_ID using Attacker's Authorization token critical
Test POST, PUT, PATCH, DELETE on Victim's object IDs — write access is higher severity critical
Try incrementing and decrementing IDs by 1, 2, 3 to check for sequential ID exposure medium
Test cross-object reference — use an order ID in a user endpoint and vice versa medium
03 — Advanced BOLA Techniques
Test BOLA in nested resources — /api/users/VICTIM_ID/orders/ORDER_ID as Attacker critical
Check batch/bulk endpoints that accept arrays of IDs: [{id: VICTIM_ID}, {id: YOUR_ID}] critical
Try adding victim's ID as an extra parameter in the request body alongside your own ID medium
Test BOLA via custom HTTP headers — X-User-ID, X-Account-ID, X-Resource-ID headers medium
Check if GUIDs/UUIDs are truly random or follow a predictable pattern (v1 UUIDs = time-based) medium
Test BOLA through search/filter endpoints — ?owner_id=VICTIM_ID or ?assigned_to=VICTIM_ID critical
Try replacing your ID with special values: *, null, 0, -1, undefined, true, false medium
04 — Auth Bypass Variations
Test endpoint with no Authorization header at all — check for completely unauthenticated access critical
Swap your JWT with Victim's token on a resource request to confirm the authorization gap critical
Try accessing victim resources with an expired token or malformed JWT signature medium
Switch HTTP methods — server may check auth on GET but forget to check on PUT or DELETE medium
Remove API version prefix — change /api/v2/users/ID to /api/v1/users/ID to hit older insecure routes medium
05 — Impact Validation & Reporting
Confirm the response actually contains Victim's data — do not stop at 200 OK alone critical
Check if the response leaks PII — name, email, phone, address, payment info, SSN, DOB critical
Test if you can modify Victim's data — BOLA + write access = critical or high severity report critical
Test if you can permanently delete Victim's objects — deletion bugs are always high impact critical
Capture the exact HTTP request and full response for your report — screenshot both easy
Reproduce the bug at least twice in a fresh session to confirm it is consistent and not a fluke easy

✅ All checks completed for IDOR!
01 — Discovery & Recon
Use the app normally as a logged-in user and capture all requests in Burp Suite History easy
Identify every place the app references a user-specific resource — profile, orders, files, invoices, messages easy
Look for numeric IDs, hashed IDs, filenames, and base64-encoded values in URLs and request bodies easy
Find file download and upload endpoints — these often expose direct path references to user files medium
Check email links and notifications — do they contain direct object references in the URL? info
Look for export and report generation endpoints — they often skip auth checks medium
Scrape JS files for hidden API routes that reference user objects not visible in the UI medium
02 — Basic IDOR Testing
Create two accounts — Attacker and Victim. Log the IDs of all objects created by the Victim easy
Replace your object ID with Victim's object ID in the URL path — /api/profile/VICTIM_ID critical
Replace your object ID in query parameters — ?user_id=VICTIM_ID and ?account=VICTIM_ID critical
Replace your object ID inside the request body — {"user_id": VICTIM_ID} in JSON POST requests critical
Test IDOR on file downloads — /download?file=invoice_125.pdf → change to invoice_124.pdf critical
Test IDOR on avatar/profile image endpoints — /api/users/VICTIM_ID/avatar medium
Try incrementing and decrementing ID values by 1, 2, 3 to access adjacent records medium
Test IDOR in pagination — ?page=2&user_id=VICTIM_ID on list endpoints medium
03 — Encoded & Hashed IDs
Decode any base64-encoded IDs and replace with Victim's base64-encoded ID medium
Try MD5 or SHA1 hash of common values — email address, username, sequential number medium
Check if UUIDs follow UUID v1 format — these are time-based and can be predicted or enumerated medium
Decode JWT payload and replace the user ID or sub claim with Victim's ID, then re-encode critical
Try URL-decoding and re-encoding encoded parameters — sometimes double-encoding bypasses checks info
Check if hashed IDs use weak or keyless hashing — HMAC with empty secret or MD5 alone medium
04 — Write & Delete IDOR
Test if you can update Victim's profile by sending a PUT request with Victim's ID critical
Try changing Victim's email or password via a POST/PUT endpoint with their ID critical
Test DELETE endpoint with Victim's object ID — can you delete their data or account? critical
Try adding items to Victim's cart, wishlist, or subscription using their ID medium
Test if you can assign tasks, tickets, or items to another user's account by changing the owner ID medium
Try submitting forms that reference Victim's ID — contact forms, support tickets, order updates medium
05 — Indirect & Chained IDOR
Test IDOR via reference fields — order has a customer_id field, change it to Victim's ID critical
Check if search/filter endpoints expose other users' data — ?owner=VICTIM_ID or ?assigned_to=VICTIM_ID medium
Test IDOR inside nested objects — {"shipping": {"user_id": VICTIM_ID}} in a checkout request medium
Try accessing shared resources via a collaborative invite link with another user's resource ID medium
Chain IDOR with mass assignment — find an object reference AND a hidden writable field together critical
Test IDOR via webhook or callback URLs that include a user's resource ID in the path info
06 — Auth & Header Bypass
Remove the Authorization header completely — test if the endpoint returns data without any token critical
Try accessing Victim's resources with an expired or revoked token medium
Test with a token from a different account type — free user accessing premium user resources critical
Try X-User-ID, X-Account-ID, or X-Forwarded-For headers with Victim's ID medium
Switch HTTP method — GET may be protected but POST or PUT on the same resource may not be medium
Test older API versions — /api/v1/ endpoints may lack the auth checks added to /api/v2/ medium
07 — Impact Validation & Reporting
Read the full response body — confirm it actually contains Victim's personal data, not just 200 OK critical
Document exactly what PII is exposed — name, email, phone, DOB, payment info, address, SSN critical
Test write and delete access — read-only IDOR is medium severity, write access is critical critical
Reproduce the bug twice in a fresh Incognito session to confirm it is consistent easy
Capture the exact HTTP request and response pair with Burp Suite for your report easy
Estimate real-world impact — how many users could be affected? Is user enumeration possible? easy

✅ All checks completed for BFLA!
01 — Recon & Function Mapping
Browse the entire app as a regular user and log every API call in Burp Suite — focus on admin-looking endpoints easy
Identify all role-sensitive functions — create, delete, approve, ban, promote, export, impersonate, refund easy
Scrape JS files for hidden admin routes — look for /admin/, /internal/, /manage/, /staff/, /superuser/ patterns medium
Check Swagger / OpenAPI docs, Postman collections, or API reference pages for privileged endpoint definitions easy
Sign up for a higher-privileged role if trial or free plan is available — map all extra functions that appear easy
Use directory brute-forcing tools (ffuf, feroxbuster) with admin-wordlists to discover hidden function endpoints medium
Check mobile app binaries (APK/IPA decompile) — they often expose admin API routes not visible in the web app medium
02 — Basic BFLA Testing
Create two accounts — a low-privilege User and a high-privilege Admin/Manager. Capture both tokens separately easy
Take every admin API call captured with the Admin token and replay it using the low-privilege User token critical
Test DELETE /api/admin/users/{id} with a regular user token — can a normal user delete any account? critical
Test POST /api/admin/users/{id}/promote or /ban or /verify with a low-privilege token critical
Test GET /api/admin/users — can a regular user list all platform users and their PII? critical
Test financial functions — issue refund, apply discount, override price, transfer credits using a regular token critical
Test content moderation endpoints — approve post, remove review, flag user, publish article as a low-privilege user medium
03 — Horizontal vs Vertical Privilege Escalation
Test vertical escalation — User role calling Admin-only functions (e.g. user calling /api/admin/config) critical
Test horizontal escalation — Manager A calling functions scoped to Manager B's team or department critical
Test cross-tenant function access — User from Org A calling admin functions scoped to Org B critical
Try calling a Moderator-only endpoint with a basic User token — e.g. /api/moderator/reports/resolve medium
Test if a deactivated or suspended account's token can still call privileged functions that active accounts cannot medium
04 — HTTP Method & Header Bypass
Switch HTTP method on restricted endpoints — if POST /admin/delete is blocked, try DELETE, PUT, PATCH on same path medium
Add X-Original-URL or X-Rewrite-URL headers pointing to /admin/ paths to bypass gateway-level access control critical
Try X-HTTP-Method-Override: DELETE or X-Method-Override: PUT header to spoof HTTP methods on firewalled endpoints medium
Add X-Role: admin, X-User-Type: superuser, or X-Privileged: true headers — some backends trust client-supplied role headers critical
Try appending /.json, /.xml, /.csv, or ?format=json to admin endpoints — format negotiation may bypass role checks medium
Test with Content-Type: application/x-www-form-urlencoded instead of JSON — some auth middleware only checks JSON requests medium
05 — API Version & Path Bypass
Downgrade API version — if /api/v2/admin/users is blocked, try /api/v1/admin/users which may lack authorization checks critical
Try path traversal on function endpoints — /api/user/../../admin/delete or /api/user/%2F%2E%2E%2Fadmin/ medium
Fuzz path capitalization and encoding — /API/Admin/Users, /api/ADMIN/users, /api/adm%69n/users to bypass case-sensitive ACLs medium
Add a trailing slash or double slash — /api/admin/users/ or /api//admin/users — routers may handle these differently medium
Test /api/mobile/admin/ or /api/internal/admin/ — mobile and internal gateways often have weaker auth than the main API critical
06 — JWT & Token Manipulation
Decode your JWT and check the payload — look for role, scope, permissions, or groups claim that could be escalated easy
Modify JWT role claim — change "role":"user" to "role":"admin" and use alg:none or resign with an empty secret critical
Add missing privilege claims — inject "is_admin":true, "scope":"admin:write", "permissions":["delete_user"] into JWT payload critical
Test with no Authorization header at all on privileged endpoints — some admin routes are completely unprotected critical
Use an expired or invalidated admin token — does the server still honour it for function-level access? medium
07 — Business-Critical Function Tests
Test bulk operations — POST /api/admin/users/bulk-delete or /bulk-update with a regular user token critical
Test system configuration endpoints — PUT /api/admin/settings, PATCH /api/config/security as a low-privilege user critical
Test user impersonation endpoints — POST /api/admin/impersonate/{user_id} — can a regular user impersonate others? critical
Test audit log access — GET /api/admin/logs — can a normal user read platform-wide security and activity logs? critical
Test webhook and integration management — create, edit or delete platform-wide webhooks as a regular user medium
Test API key management — POST /api/admin/apikeys/generate or revoke another user's API key as a low-privilege account critical
08 — Impact Validation & Reporting
Confirm the action actually executed — don't stop at 200 OK, verify the side effect (user deleted, role changed, data modified) critical
Rate the severity correctly — unauthorized admin READ is high, unauthorized admin WRITE/DELETE is critical critical
Document the exact role gap — state clearly which role called which function and what role is required to call it medium
Reproduce the bug in a clean session with only the low-privilege token — rule out any session bleed from the admin account easy
Capture the full HTTP request and response with Burp Suite — include both the low-priv token and the admin-level response easy
Estimate business impact — platform takeover? Mass data leak? Financial loss? User enumeration? Scale the severity accordingly easy

✅ All checks completed for Authentication!
01 — Recon & Attack Surface Mapping
Map all authentication endpoints — /login, /register, /logout, /token, /refresh, /forgot-password, /reset-password, /verify, /2fa easy
Identify which auth mechanism is used — JWT, session cookie, API key, OAuth2, Basic Auth, or a combination easy
Check if the app exposes auth endpoints on multiple paths — /api/v1/auth, /api/v2/auth, /mobile/auth, /internal/auth easy
Look for third-party OAuth providers — Google, GitHub, Facebook — and map all OAuth callback and redirect endpoints easy
Scrape JS files and mobile binaries for hardcoded API keys, tokens, client secrets, or credentials left in source code medium
Check error messages on login — do they reveal whether the username exists vs wrong password? (User enumeration) medium
02 — Login & Brute Force Testing
Test for missing rate limiting on /login — send 100+ requests with wrong passwords and check if account locks or requests get blocked critical
Test credential stuffing resilience — try common username/password combos (admin:admin, test:test, user:password) on login medium
Check if rate limiting can be bypassed using X-Forwarded-For, X-Real-IP, or X-Originating-IP headers to rotate IP address critical
Test if CAPTCHA on login can be bypassed — reuse a solved CAPTCHA token, omit it entirely, or use an old token repeatedly medium
Test login with SQL injection payloads — ' OR '1'='1, admin'-- in the username field to attempt auth bypass critical
Test login with NoSQL injection — {"username": {"$gt": ""}, "password": {"$gt": ""}} for MongoDB-backed auth endpoints critical
Check if account lockout policy exists — after N failed attempts does it lock, and can lockout itself be abused to DoS other users? medium
03 — JWT Security Testing
Decode the JWT at jwt.io — inspect all claims: sub, role, exp, iat, iss, aud — look for sensitive data stored in the payload easy
Test alg:none attack — remove the signature, set "alg":"none" in the header and send the unsigned token to a protected endpoint critical
Test RS256 → HS256 confusion attack — take the RS256 public key and use it as the HMAC secret to sign a forged HS256 token critical
Test weak JWT secret — brute-force the HMAC secret using hashcat or jwt-cracker with rockyou.txt wordlist critical
Test JWT header injection — inject "jwk" or "jku" or "x5u" claim pointing to your own server to supply a forged public key critical
Check if expired JWTs are still accepted — change exp claim to a past timestamp and test if the server validates expiry correctly critical
Check if JWT is invalidated on logout — after logout, replay the old token and verify the server rejects it with 401 critical
Test kid (Key ID) header injection — set "kid":"../../dev/null" or inject SQL into the kid claim to manipulate key lookup critical
04 — Token & Session Management
Check token entropy — are API keys or session tokens short, sequential, or predictable? Try generating 10 tokens and compare medium
Test refresh token reuse — after a token refresh, can the old refresh token still be used again to get new access tokens? critical
Check if tokens are sent over HTTP (not HTTPS) — inspect any non-TLS requests that carry auth tokens in headers or cookies critical
Check session cookie flags — confirm HttpOnly, Secure, and SameSite=Strict/Lax are all set on auth cookies medium
Test token leakage in URL — check if the access token is passed as a query param (?token=...) which gets logged in server logs medium
Test concurrent session handling — login from two different devices and check if both sessions remain active or the older one is revoked info
05 — Password Reset & OTP Testing
Test password reset token expiry — request a reset link, wait 1 hour, and check if the token is still valid and usable medium
Test if password reset token is invalidated after use — reuse the same reset link a second time and check if it still works critical
Test Host header injection on password reset — change the Host header to your server and check if the reset link points to your domain critical
Test OTP brute force — send 10,000 requests cycling all possible 4 or 6-digit OTP values with no rate limiting or lockout critical
Test OTP reuse — successfully verify an OTP, then immediately try to use the same OTP again on a second request critical
Test password reset token predictability — generate 5 reset tokens and analyze if they follow a pattern (timestamp-based, sequential) medium
Test reset token scope — can you use your own valid reset token to reset a different user's password by swapping the email param? critical
06 — OAuth2 & SSO Testing
Test OAuth redirect_uri validation — change redirect_uri to your own domain and check if the auth code is sent to your server critical
Test CSRF on OAuth flow — check if the state parameter is present, validated, and tied to the user's session on callback critical
Test OAuth authorization code reuse — intercept and replay the auth code after it has already been exchanged for a token critical
Test account takeover via OAuth email trust — register with a social provider using Victim's email if the app trusts unverified emails critical
Check open redirect in OAuth flow — test if redirect_uri accepts wildcards, subdomains, or path traversal to redirect to attacker's domain critical
Test scope escalation — request a token with minimal scopes, then try using it to call APIs that require higher scopes medium
07 — Multi-Factor Authentication Bypass
Test if MFA can be skipped entirely — after entering valid credentials, directly call authenticated endpoints without completing the MFA step critical
Test MFA code response manipulation — intercept the MFA verification response and change "success":false to "success":true critical
Test if backup codes are rate limited — try brute-forcing all possible backup codes if they are short numeric values critical
Test if MFA is enforced on all login paths — check /api/mobile/login, /api/v1/login, /social/login for endpoints that skip MFA critical
Test TOTP code reuse — use a valid TOTP code, wait 1 second, and try the same code again before the 30-second window expires medium
08 — Impact Validation & Reporting
Confirm full account takeover is achievable — can you log in as Victim, change their password, and lock them out permanently? critical
Document exact steps to reproduce — include every request, header, token value, and response that proves the vulnerability easy
Rate severity accurately — user enumeration is low/info, JWT none attack is critical, MFA bypass is critical medium
Reproduce in a clean Incognito session using only the attacker account — eliminate any admin cookie or session bleed easy
Capture the exact HTTP request and full response with Burp Suite — clearly mark which token was used and what it should not have accessed easy
Estimate blast radius — single account takeover, mass account takeover, or full platform compromise? State user count at risk easy

✅ All checks completed for Business Logic!
01 — Workflow & Process Abuse
Skip steps in a multi-step workflow — go directly from step 1 to step 3 by calling the final API endpoint without completing intermediate steps critical
Replay an old valid request from an earlier workflow step after the workflow has already completed — check if the server accepts it again critical
Complete a workflow in reverse order — call the confirmation endpoint before the initiation endpoint and observe server behavior medium
Interrupt a payment or checkout flow mid-way and then access the confirmation/success endpoint directly — check if order is placed without payment critical
Test if the server enforces state transitions on the backend — or if it only relies on the frontend to control which step the user is on critical
Submit the same workflow step multiple times in parallel using Burp Suite Repeater — check if race conditions allow double processing critical
02 — Price & Discount Manipulation
Intercept the checkout request and modify the price or total_amount field to 0, 1, or a negative value — check if the server accepts the client-supplied price critical
Apply the same discount coupon code multiple times in a single order or across multiple orders — check if single-use coupons are enforced server-side critical
Apply multiple different discount codes or offers to a single cart — check if stacking is allowed beyond intended business rules medium
Change the item quantity to a negative number in the cart — check if a negative quantity subtracts from the total, resulting in a lower or zero total price critical
Add a high-value item with a coupon, then swap the item for a different item after the discount is applied — check if the discount is re-validated medium
Test currency confusion — submit a price in a different currency code (USD vs EUR) and observe if the server converts or blindly accepts it medium
03 — Race Conditions
Send multiple simultaneous requests to redeem a single-use coupon or gift card — use Burp Suite Turbo Intruder or the parallel requests feature to send all at once critical
Race condition on account balance — send two simultaneous withdrawal requests for the full balance and check if both succeed, withdrawing double the amount critical
Race condition on inventory — buy the last item in stock with two simultaneous purchase requests — check if both orders are confirmed despite only one item available medium
Race condition on referral or cashback systems — trigger the reward event simultaneously from two sessions to see if the bonus is credited twice critical
Race condition on OTP or token validation — send parallel validation requests with the same OTP before it is marked as used on the server critical
04 — Limit & Restriction Bypass
Exceed the maximum allowed quantity per item in a purchase — change qty to 9999 and see if the server enforces the limit or processes the inflated order medium
Bypass free trial limits — create a new account after the trial expires and link back to the same payment method or device to get another free trial medium
Access premium features with a free or basic plan account — directly call premium API endpoints without upgrading the subscription critical
Test upload limits — upload a file exceeding the max size limit by manipulating the Content-Length header or using chunked transfer encoding medium
Test daily/monthly API usage limits — check if limits reset properly or if the counter can be manipulated via parameter tampering medium
Test referral program limits — refer yourself using a different email or phone number to earn referral bonuses on your own account medium
05 — Input Validation Logic Flaws
Submit extreme numeric values — very large integers (Integer.MAX_VALUE), very small decimals (0.0001), or negative numbers in fields that should only accept positives medium
Test data type confusion — submit a string where a number is expected, an array where a string is expected, or a boolean where an ID is expected medium
Test boundary values — submit exactly the maximum allowed value, then one above and one below to find off-by-one logic errors in the backend medium
Submit empty or null values for required fields — check if the server throws verbose errors, skips validation, or processes the request with missing data easy
Test integer overflow — submit a value like 2147483648 (INT_MAX + 1) and check if the backend wraps it to a negative number, bypassing a minimum balance check critical
Submit Unicode or special characters in name and address fields — check if they pass validation on one endpoint but fail on another, creating inconsistency exploits info
06 — Trust & Role Assumption Flaws
Check if the server trusts user-supplied role or permission fields — add "role": "admin" or "is_admin": true to a registration or profile update request critical
Manipulate the account type or plan field in an API request — change "plan": "free" to "plan": "enterprise" and check if features are unlocked critical
Test if the application trusts client-supplied IP addresses — add X-Forwarded-For: 127.0.0.1 to bypass IP-based restrictions or internal-only endpoint access critical
Check if email verification is enforced before granting account privileges — register with an unverified email and test if you can access all features immediately medium
Test if KYC or identity verification can be bypassed by skipping the verification step and directly accessing post-verification API endpoints critical
07 — Refund, Return & Financial Logic
Request a refund for an order and then also dispute the charge with the payment provider — check if the system allows double refund without flagging critical
Submit multiple refund requests for the same order simultaneously — race condition on refund endpoint to receive the refund amount more than once critical
Test if refunds can be sent to a different payment method or account than the original — manipulate the refund destination in the refund API request critical
Check if wallet or credit balance can go negative — withdraw more than the available balance by manipulating the amount field or sending parallel withdrawals critical
Test loyalty or reward points — check if points are credited before a purchase is confirmed, allowing points to be earned from cancelled or refunded orders medium
08 — Impact Validation & Reporting
Confirm the bug causes a real, demonstrable financial or functional impact — not just an unexpected response but an actual gain, data access, or workflow bypass critical
Reproduce the bug at least twice in a clean session to confirm consistency — business logic bugs can be environment-sensitive easy
Document the exact steps, request sequence, and manipulated parameters — business logic reports need a very clear step-by-step reproduction to be accepted easy
Calculate and state the maximum financial impact in your report — how much money could be stolen, how many accounts affected, what is the worst-case scenario easy
